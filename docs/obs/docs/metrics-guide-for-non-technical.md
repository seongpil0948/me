## 📊 우리가 측정하는 지표들

### 1. 사용자 활동 지표

#### 👥 MAU (Monthly Active Users) - 월간 활성 사용자

**무엇을 측정하나요?**

- 한 달 동안 서비스를 1회 이상 방문한 고유 사용자 수

**데이터 수집 방식:**

```
수집 이벤트: commerce.session.start (세션 시작)
집계 기준: 사용자 ID (attributes.userKey) 기준 중복 제거
필터 조건:
  - 특정 월(year-month)의 모든 로그 조회
  - userKey가 NULL이 아닌 정상 로그만 집계
  - 서비스별 분리 (theshop-brand, theshop-pharmacy 등)

계산식:
  COUNT(DISTINCT attributes.userKey)
  WHERE body = 'commerce.session.start'
    AND year-month = '2025-10'
    AND attributes.userKey IS NOT NULL
```

**어떻게 계산하나요?**

- 매월 1일부터 말일까지 한 번이라도 `commerce.session.start` 이벤트를 발생시킨 고유 사용자(userKey)를 집계합니다
- 같은 사람(동일 userKey)이 100번 방문해도 1명으로 계산됩니다
- 앱 실행, 웹사이트 접속 시 자동으로 세션 시작 이벤트가 기록됩니다

**왜 중요한가요?**

- 서비스의 전체적인 규모와 성장세를 파악할 수 있습니다
- 마케팅 캠페인 효과를 장기적으로 측정할 수 있습니다

**실제 활용 예시:**

```
1월 MAU: 15,000명
2월 MAU: 18,000명 (👆 20% 증가)
→ 설날 프로모션이 신규 유입에 효과적이었음
```

**대시보드에서 보는 방법:**

- "핵심 성과 지표" 섹션의 "이번 달 MAU" 카드
- "월별 활성 사용자 추이" 그래프 (12개월 바 차트)

#### 📅 DAU (Daily Active Users) - 일간 활성 사용자

**무엇을 측정하나요?**

- 하루 동안 서비스를 방문한 고유 사용자 수

**데이터 수집 방식:**

```
수집 이벤트: commerce.session.start
집계 기준: 날짜(year-month-day) + 사용자 ID 기준 중복 제거
필터 조건:
  - 최근 90일간의 일별 로그 조회
  - 파티션 필터링으로 성능 최적화
    (year, month, day 컬럼 활용)

계산식:
  일별로 GROUP BY 후
  COUNT(DISTINCT attributes.userKey)
  WHERE body = 'commerce.session.start'
    AND date = '2025-10-29'
```

**어떻게 계산하나요?**

- 매일 00:00:00 ~ 23:59:59 동안 `commerce.session.start` 이벤트를 발생시킨 고유 사용자를 일별로 집계합니다
- MAU와 마찬가지로 동일 사용자의 중복 방문은 1명으로 계산됩니다
- 로그 데이터의 year/month/day 파티션을 활용하여 빠르게 조회합니다

**왜 중요한가요?**

- 일별 트렌드와 요일별 패턴을 파악할 수 있습니다
- 특정 이벤트나 프로모션의 즉각적인 효과를 측정할 수 있습니다

**실제 활용 예시:**

```
평일 평균 DAU: 2,500명
주말 평균 DAU: 3,200명 (28% 높음)
→ 주말에 주문이 집중되므로 금요일 저녁 푸시 알림 발송 전략
```

**대시보드에서 보는 방법:**

- "핵심 성과 지표" 섹션의 "어제 DAU" 카드
- "일일 활성 사용자 추이" 라인 차트 (최근 30일)

### 2. 사용자 충성도 지표

#### 🔄 Retention (리텐션) - 재방문율

**무엇을 측정하나요?**

- 특정 날짜에 방문한 사용자가 며칠 후에 다시 돌아오는 비율

**데이터 수집 방식:**

```
수집 이벤트: commerce.session.start
집계 기준: 사용자 ID 기반 날짜별 방문 기록 추적

단계별 로직:

1. 일별 방문 사용자 추출 (최근 45일)
   SELECT DISTINCT
     CAST(year || '-' || month || '-' || day AS DATE) as visit_date,
     attributes.userKey as user_key
   FROM logall
   WHERE body = 'commerce.session.start'
     AND attributes.userKey IS NOT NULL
   GROUP BY visit_date, user_key

2. 기준일(base) 방문자와 비교일(target) 방문자 LEFT JOIN

   D+1 리텐션 예시:
   - base: 2일 전(10/27) 방문자 집합
   - target: 1일 전(10/28) 방문자 집합
   - JOIN: base.user_key = target.user_key

3. 리텐션율 계산
   retention_rate =
     (재방문한 사용자 수 ÷ 기준일 방문자 수) × 100

   COUNT(DISTINCT CASE WHEN target.user_key IS NOT NULL
                       THEN base.user_key END) /
   COUNT(DISTINCT base.user_key) * 100
```

**어떻게 계산하나요?**
우리는 3가지 리텐션을 측정합니다:

1. **D+1 리텐션** (다음날 재방문율)
   - **기준일**: 실행일 기준 2일 전 (10/27)
   - **비교일**: 실행일 기준 1일 전 (10/28)
   - **로직**: 10/27에 `commerce.session.start` 이벤트를 발생시킨 사용자(userKey) 중, 10/28에도 동일 이벤트를 발생시킨 사용자의 비율
   - 예: 10/27 방문자 100명 → 10/28 재방문 35명 = 35%

2. **D+7 리텐션** (일주일 후 재방문율)
   - **기준일**: 실행일 기준 8일 전 (10/20)
   - **비교일**: 실행일 기준 1일 전 (10/27)
   - **로직**: 10/20 방문자 중 10/27에 재방문한 비율
   - 예: 10/20 방문자 100명 → 10/27 재방문 22명 = 22%

3. **D+30 리텐션** (한달 후 재방문율)
   - **기준일**: 실행일 기준 31일 전 (9/27)
   - **비교일**: 실행일 기준 1일 전 (10/27)
   - **로직**: 9/27 방문자 중 10/27에 재방문한 비율
   - 예: 9/27 방문자 100명 → 10/27 재방문 8명 = 8%

**중요:**

- 세션(sessionId)이 아닌 **사용자(userKey) 단위**로 추적합니다
- 비교일(target)에 여러 번 방문해도 1명으로 계산됩니다
- LEFT JOIN을 사용하여 기준일 방문자가 재방문하지 않은 경우도 포함합니다

**왜 중요한가요?**

- 서비스 만족도와 습관 형성 정도를 알 수 있습니다
- 신규 가입자가 정착하는지 이탈하는지 파악할 수 있습니다

**건강한 지표는?**

```
D+1 리텐션: 30-40% 이상 → 우수
D+7 리텐션: 15-25% 이상 → 양호
D+30 리텐션: 5-10% 이상 → 정상
```

**실제 활용 예시:**

```
신규 회원 가입 캠페인 후:
- D+1 리텐션: 25% (😟 목표 35% 미달)
- 개선 액션: 가입 다음날 웰컴 할인 쿠폰 발송
- 결과: D+1 리텐션 38%로 상승 ✅
```

**대시보드에서 보는 방법:**

- "핵심 성과 지표"의 "최신 D+1 리텐션" 테이블
- "리텐션 및 전환 분석" 섹션의 "최신 리텐션 비율" 상세 테이블

#### 🎯 Conversion (전환율) - 구매 전환율

**무엇을 측정하나요?**

- 방문자가 실제 구매 행동으로 이어지는 비율

**데이터 수집 방식:**

```
수집 이벤트 (3가지):
  1. commerce.session.start    (세션 시작)
  2. commerce.product.view     (상품 조회)
  3. commerce.cart.add         (장바구니 담기)

집계 기준: 세션 ID (attributes.sessionId) 단위

단계별 로직:

1. 일별 이벤트 데이터 추출 (최근 90일)
   SELECT
     date,
     attributes.sessionId,
     attributes.userKey,
     body  -- 이벤트 타입
   FROM logall
   WHERE body IN ('commerce.session.start',
                  'commerce.product.view',
                  'commerce.cart.add')
     AND attributes.sessionId IS NOT NULL
   GROUP BY date, sessionId, userKey, body

2. 세션 단위로 이벤트 발생 여부 집계
   - total_sessions:
       session.start 이벤트가 있는 고유 세션 수
   - product_view_sessions:
       product.view 이벤트가 있는 고유 세션 수
   - cart_add_sessions:
       cart.add 이벤트가 있는 고유 세션 수

   COUNT(DISTINCT CASE WHEN body = 'commerce.session.start'
                       THEN sessionId END)

3. 전환율 계산
   cart_conversion_rate =
     (cart_add_sessions ÷ product_view_sessions) × 100
```

**어떻게 계산하나요?**
우리는 구매 퍼널을 3단계로 나눠서 **세션(sessionId) 기준**으로 측정합니다:

```
1단계: 방문 (세션 시작)
   ↓ 제품 조회율
2단계: 상품 조회
   ↓ 장바구니 전환율
3단계: 장바구니 담기
```

**주요 지표들:**

1. **상품 조회율** = (상품을 본 세션 수 ÷ 전체 세션 수) × 100
   - 분자: `commerce.product.view` 이벤트가 1회 이상 발생한 고유 세션 수
   - 분모: `commerce.session.start` 이벤트가 발생한 전체 세션 수

2. **장바구니 전환율** = (장바구니 담은 세션 수 ÷ 상품 조회 세션 수) × 100
   - 분자: `commerce.cart.add` 이벤트가 1회 이상 발생한 고유 세션 수
   - 분모: `commerce.product.view` 이벤트가 발생한 세션 수

3. **세션 전환율** (overall_conversion_rate) = (장바구니 담은 세션 수 ÷ 전체 세션 수) × 100
   - 분자: `commerce.cart.add` 이벤트가 발생한 세션 수
   - 분모: `commerce.session.start` 이벤트가 발생한 전체 세션 수
   - **대시보드 표시**: "최근 세션 전환율" 게이지, "세션 전환율 추이" 차트

**중요:**

- 모든 전환율 지표는 **세션 단위** 집계입니다
- 같은 세션에서 여러 번 이벤트 발생해도 1개 세션으로 계산
- 한 세션 내에서 상품을 10개 담아도 "장바구니 담은 세션 1개"로 카운트
- "장바구니 담은 사용자 수" 컬럼은 고유 사용자(userKey) 기준이지만, 전환율 계산은 세션 기준

**왜 중요한가요?**

- 어느 단계에서 고객이 이탈하는지 정확히 파악할 수 있습니다
- 각 단계별로 최적화 전략을 세울 수 있습니다

**건강한 지표는?**

```
전체 전환율: 3-8% → 이커머스 평균
장바구니 전환율: 10-20% → 목표 수준
```

**실제 활용 예시:**

```
현재 상황:
- 상품 조회율: 70% ✅ 양호
- 장바구니 전환율: 8% 😟 낮음

분석:
→ 고객이 상품은 보지만 장바구니에 담지 않음

개선 액션:
→ 상품 상세 페이지에 "지금 담으면 5% 할인" 뱃지 추가
→ 장바구니 전환율 12%로 상승
```

**대시보드에서 보는 방법:**

- "핵심 성과 지표" 섹션의 "최근 세션 전환율" 게이지
- "세션 전환율 추이 (30일)" 라인 차트
- "종합 일일 대시보드" 테이블에서 일별 상세 데이터

### 3. 사용자 세분화 지표

#### 👤 User Cohort (사용자 코호트) - 신규/활성/휴면 분류

**무엇을 측정하나요?**

- 첫 방문 시점을 기준으로 사용자를 4개 그룹으로 분류

**어떻게 분류하나요?**

| 그룹명               | 첫 방문 시점  | 특징                    |
| -------------------- | ------------- | ----------------------- |
| **New (신규)**       | 최근 7일 이내 | 처음 가입한 신규 회원   |
| **Recent (최근)**    | 8-30일 전     | 정착 단계의 회원        |
| **Active (활성)**    | 31-90일 전    | 습관이 형성된 충성 고객 |
| **Veteran (베테랑)** | 90일 이전     | 장기 고객               |

**왜 중요한가요?**

- 그룹별로 다른 마케팅 전략을 적용할 수 있습니다
- 신규 회원이 활성 회원으로 전환되는 과정을 추적할 수 있습니다

**실제 활용 예시:**

```
현재 분포:
- New: 1,200명 (20%)
- Recent: 800명 (13%)
- Active: 2,500명 (42%)
- Veteran: 1,500명 (25%)

맞춤 전략:
- New → 온보딩 튜토리얼, 첫 구매 할인 쿠폰
- Recent → 재방문 유도 푸시 알림
- Active → VIP 프로그램 안내
- Veteran → 신상품 우선 공개
```

**대시보드에서 보는 방법:**

- "사용자 세분화 및 상세 분석" 섹션의 코호트 분포 차트

### 4. 세션 품질 지표

#### 🔢 Session Metrics (세션 메트릭) - 방문당 활동량

**무엇을 측정하나요?**

- 한 번 방문했을 때 얼마나 적극적으로 활동하는지

**주요 지표들:**

1. **평균 이벤트 수/세션**
   - 한 번 방문시 평균 몇 개의 행동을 하는지
   - 예: 상품 조회, 검색, 장바구니 담기 등

2. **평균 이벤트 타입 수/세션**
   - 한 번 방문시 몇 가지 다양한 활동을 하는지

3. **중앙값 이벤트 수**
   - 평균이 아닌 중간값 (극단값 영향 제거)

4. **상위 10% 세션**
   - 가장 활발한 10%는 얼마나 많은 활동을 하는지

**왜 중요한가요?**

- 사용자 참여도(Engagement)를 정량적으로 측정할 수 있습니다
- 새로운 기능이 참여도를 높이는지 확인할 수 있습니다

**건강한 지표는?**

```
평균 이벤트/세션: 10-20개 → 정상
평균 이벤트 타입/세션: 4-6개 → 다양한 활동
```

**실제 활용 예시:**

```
신규 기능 "빠른 주문" 출시 전:
- 평균 이벤트/세션: 12개

출시 후:
- 평균 이벤트/세션: 15개 (25% 증가)
→ 기능이 사용자 참여도를 높였음을 확인
```

### 5. 이벤트 분포 지표

#### 📈 Event Distribution (이벤트 분포) - 행동 패턴 분석

**무엇을 측정하나요?**

- 사용자들이 어떤 행동을 가장 많이 하는지

**어떻게 측정하나요?**

- 최근 7일간 발생한 모든 이벤트를 종류별로 집계
- 각 이벤트가 전체에서 차지하는 비율 계산

**주요 이벤트 종류:**

- `commerce.session.start`: 앱/웹 접속
- `commerce.product.view`: 상품 조회
- `commerce.cart.add`: 장바구니 담기
- `commerce.search`: 검색
- `commerce.button.click`: 버튼 클릭
- `commerce.checkout.purchase`: 구매 완료

**왜 중요한가요?**

- 사용자가 주로 하는 활동을 파악할 수 있습니다
- 특정 기능의 사용률 변화를 모니터링할 수 있습니다

**실제 활용 예시:**

```
이벤트 분포 (최근 7일):
1. commerce.session.start: 45%
2. commerce.product.view: 30%
3. commerce.cart.add: 15%
4. commerce.search: 8%
5. commerce.checkout.purchase: 2%

인사이트:
→ 검색 사용률이 낮음 (8%)
→ 검색 UI를 더 눈에 띄게 개선 필요
```

## 🏪 커머스 지표들

우리 서비스만의 특별한 커머스 관련 지표입니다.

### 1. 버튼 클릭 분석

**무엇을 측정하나요?**

- 어떤 버튼/기능이 가장 많이 사용되는지

**데이터 수집 방식:**

```
수집 이벤트: commerce.button.click
집계 기준: 버튼 속성별 그룹화
수집 기간: 최근 7일

로그 구조:
  body = 'commerce.button.click'
  attributes.action     -- 버튼 행동 (예: smart_order_open)
  attributes.menuName   -- 메뉴명
  attributes.page       -- 페이지 위치
  attributes.userKey    -- 사용자 ID
  attributes.sessionId  -- 세션 ID

계산식:
  GROUP BY action, menuName, page
  총 클릭 수: COUNT(*)
  고유 사용자: COUNT(DISTINCT userKey)
  평균 클릭/사용자: COUNT(*) ÷ COUNT(DISTINCT userKey)
```

**주요 지표:**

- **총 클릭 수**: `commerce.button.click` 이벤트 발생 총 횟수 (중복 포함)
- **고유 사용자 수**: 해당 버튼을 1회 이상 클릭한 고유 사용자(userKey) 수
- **평균 클릭 수/사용자**: 사용자 1명당 평균 몇 번 클릭했는지

**실제 활용 예시:**

```
스마트주문 버튼:
- 총 클릭: 1,500회
- 고유 사용자: 500명
- 평균 클릭/사용자: 3회
→ 사용자 1명당 평균 3번 재사용하는 인기 기능
```

### 2. 상품 랭킹

**무엇을 측정하나요?**

- 어떤 상품이 가장 인기있는지 (장바구니 담기 기준)

**데이터 수집 방식:**

```
수집 이벤트: commerce.cart.add
집계 기준: 상품명별 그룹화
수집 기간: 최근 30일
TOP N: 상위 100개 상품

로그 구조:
  body = 'commerce.cart.add'
  attributes.productName  -- 상품명
  attributes.category     -- 카테고리 (또는 ctgnm)
  attributes.sellerNm     -- 판매자명
  attributes.quantity     -- 수량
  attributes.salePrice    -- 판매가 (또는 price)
  attributes.userKey      -- 사용자 ID
  attributes.sessionId    -- 세션 ID

계산식:
  GROUP BY productName, category, sellerNm

  장바구니 담기 횟수: COUNT(*)
  고유 사용자: COUNT(DISTINCT userKey)
  총 수량: SUM(TRY_CAST(quantity AS INTEGER))
  평균 가격: AVG(TRY_CAST(salePrice AS DOUBLE))
  총 장바구니 가치:
    SUM(quantity × salePrice)

  ORDER BY COUNT(*) DESC
  LIMIT 100
```

**주요 지표:**

- **장바구니 담기 횟수**: `commerce.cart.add` 이벤트 발생 총 횟수 (같은 상품 여러 번 담은 것도 모두 카운트)
- **고유 구매자 수**: 해당 상품을 장바구니에 담은 고유 사용자(userKey) 수
- **평균 구매 단가**: 해당 상품의 평균 판매가 (attributes.salePrice 또는 attributes.price)
- **총 장바구니 가치**: (수량 × 가격)의 합계

**참고:**

- 동일 상품도 세션이 다르거나 시점이 다르면 별도 카운트

**실제 활용 예시:**

```
Top 3 상품 (최근 30일):
1. 비타민 C 1000mg - 장바구니 담기 850회
2. 오메가3 - 장바구니 담기 720회
3. 프로바이오틱스 - 장바구니 담기 650회

→ 메인 페이지에 이 3개 상품 우선 노출
```

- 아침 9시: 대시보드에서 전날 데이터 확인 가능

### 3단계: 대시보드 조회

```
Grafana 대시보드 → 실시간 업데이트 → 시각화된 차트
```

## 💡 실전 활용 시나리오

### 시나리오 1: 신규 가입 프로모션 효과 측정

**확인할 지표:**

1. **MAU/DAU 증가율**
   - 프로모션 전후 비교
   - 목표: 20% 이상 증가

2. **D+1 리텐션**
   - 신규 가입자가 다음날 재방문하는지
   - 목표: 35% 이상

3. **User Cohort 분포**
   - "New" 그룹 비중 증가 확인

**판단 기준:**

```
✅ 성공: MAU +25%, D+1 리텐션 40%
😐 보통: MAU +15%, D+1 리텐션 30%
❌ 실패: MAU +5%, D+1 리텐션 20%
```

**의사결정 프로세스:**

```
1. 상품 랭킹 확인 → Top 10 선정

3. 시간대 분석 → 저녁 7시 푸시 알림 발송

```

### 시나리오 3: 사용자 이탈 방지

**질문:** "왜 신규 사용자가 떠나나요?"

**확인할 지표:**

1. **Retention by Cohort**
   - New 그룹의 D+1 리텐션이 낮은가?

2. **Session Metrics**
   - 신규 사용자의 평균 이벤트 수가 낮은가?

3. **Conversion Funnel**
   - 어느 단계에서 이탈이 많은가?

**개선 액션:**

```
문제: D+1 리텐션 20% (목표 35%)

원인 파악:
- 첫 방문시 평균 이벤트 5개 (전체 평균 12개)
- 상품 조회율 30% (전체 평균 70%)
→ 첫 방문시 목적을 찾지 못하고 이탈

해결책:
1. 신규 가입시 관심사 설문 추가
2. 맞춤 상품 추천 강화
3. 가입 다음날 웰컴 쿠폰 발송

결과 측정:
- 2주 후 D+1 리텐션 재확인
- 목표: 35% 이상 달성
```

## 🎯 지표별 액션 가이드

### MAU/DAU가 감소하면?

**체크리스트:**

- [ ] 경쟁사 프로모션 확인
- [ ] 앱 오류/버그 확인
- [ ] 푸시 알림 발송 여부 확인
- [ ] 계절적 요인 고려 (명절, 휴가 등)

**즉시 액션:**

1. 재방문 유도 이벤트 기획
2. 휴면 회원 타겟 푸시 발송
3. SNS 마케팅 강화

---

### Retention이 낮으면?

**체크리스트:**

- [ ] 신규 사용자 온보딩 과정 점검
- [ ] 첫 구매 유도 혜택 충분한지
- [ ] 경쟁사 대비 서비스 차별점 명확한지

**즉시 액션:**

1. 가입 후 3일간 매일 푸시 알림 (자동화)
2. 첫 구매 할인율 상향 조정
3. 사용 가이드 콘텐츠 제작

---

### Conversion이 낮으면?

**단계별 진단:**

**1단계 (방문→상품조회) 문제**

- 메인 페이지 개선
- 추천 상품 알고리즘 최적화

**2단계 (상품조회→장바구니) 문제**

- 상품 상세 페이지 개선
- 가격 경쟁력 점검
- 리뷰/평점 강화

**3단계 (장바구니→구매) 문제**

- 결제 프로세스 간소화
- 배송비/결제 수단 개선
- 장바구니 리마인드 푸시

## 🔍 FAQ

### Q1. 대시보드 데이터는 언제 업데이트되나요?

**A.** 매일 새벽 2시-4시에 자동 업데이트됩니다. 아침 9시 이후에는 전날 데이터를 확인할 수 있습니다.

### Q2. 실시간 데이터를 볼 수 없나요?

**A.** 현재는 일일 집계만 지원합니다. 실시간이 꼭 필요하다면 데이터팀에 문의해주세요.

### Q3. 왜 MAU가 DAU보다 작게 나오나요?

**A.** 버그입니다! 즉시 데이터팀에 알려주세요. 정상적으로는 MAU ≥ DAU입니다.

### Q4. 특정 캠페인의 효과만 따로 볼 수 있나요?

**A.** 기본 대시보드에는 없습니다. 커스텀 분석을 요청해주세요.

### Q5. 경쟁사와 비교 데이터도 있나요?

**A.** 현재는 자사 데이터만 수집합니다. 산업 평균 벤치마크는 외부 리포트를 참고해주세요.
